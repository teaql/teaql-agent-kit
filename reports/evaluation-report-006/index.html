阶段四：根因复盘（关键发现）

💡 核心教训：没有用 generate-lib 的标准初始化流程

生成的 runtime.rs 提供了 service_runtime_from_env() 函数， 它自动完成：读环境变量 → 连接数据库 → ensure_schema（建表 + initial_graph 种子） → 注册所有组件。 但 AGENTS.md 只教了手动拼装 UserContext::new().with_module()... 的底层路径， 导致 Agent 绕过了标准入口。

三个"修复"的重新评估：

修复操作根因评价
删除 initial_graph 手动 INSERT + initial_graph 并行执行 → UNIQUE 冲突 不应该删 initial_graph 是常量标准数据源。走 ensure_schema 不会有冲突
加 teaql-runtime 依赖 generate-lib 没有 pub re-export teaql_runtime 合理但可以绕过 如果不写类型注解可不用显式导入
加 AuditedSave import save() 需要 trait scope 取决于调用方式 某些场景可隐式使用

文档修复验证（18:07）：

重新运行 teaql gen-lib（teaql 1.0.0），新 AGENTS.md 已包含：
## Runtime Setup Use the generated runtime helper as the default application entrypoint: let ctx = school_service::service_runtime_from_env().await?;

并明确警告："Do not hand-write duplicate seed INSERT statements for generated constants."

透明跟踪与审计分析

TeaQL 的一个核心设计特色是透明的跟踪与审计系统。 本次评估专门验证了这个系统的实际效果。

TeaQL 三层审计防御

第一层：类型系统

save() 是 pub(crate) — 外部 crate 无法直接调用。 必须通过 repository API 或 .audit_as().save() 链。 编译期即强制。

第二层：参数强制

外部写入路径必须：repo.save_entity_with_comment(entity, status, "comment")。 comment 参数是强制的，系统会将其写入审计日志。没有 comment，无法保存。

第三层：语义约束

FK 字段不能直接设 ID（如 update_platform_id()）， 必须用关联方法 update_platform_to_xxx()。 常量实体在 i≥200 后冻结，不可更改。

安全审计实验（2026-06-13 15:51）

人工验证了外部 crate 无法绕过审计直接调用 save()：

// 外部 crate ❌ 不能直接调用 save() entity.save(&ctx).await?; // 编译错误：save() 是 pub(crate) // 外部 crate ✅ 正确路径 let repo = ctx.doc_entry_repository()?; repo.save_entity_with_comment(entry, EntityStatus::New, "意图描述").await?; // FK ❌ 不能直接设 ID entity.update_platform_id(1); // 不存在这个方法 // FK ✅ 必须用关联方法 entity.update_platform_to_default();

审计对评分的影响

审计系统的存在直接影响了 审计与追踪 维度的评分。 TeaQL 的审计设计（类型系统 → 参数强制 → 语义约束）使得 Agent 的输出即使有误， 也容易被审计链发现和纠正。这提升了该维度得分。

但如果 Agent 早期就充分利用 TEAQL_SQL 日志进行调试， 可以更快定位 initial_graph 冲突问题，减少不必要的绕路。 这一点在评分中有所反映（项目理解 8.0 而非满分）。
