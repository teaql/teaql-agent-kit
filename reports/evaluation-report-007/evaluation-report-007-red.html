⚡Token 节省分析

TeaQL 的核心设计理念之一是 「能量意识」——避免不必要的计算、注意力和 Token 消耗。在 AI Agent 开发中，这直接体现为可量化的 Token 节省。

📄
样板代码消除
50 行的 KSML 模型 → 24 个 Rust 源文件 (3,500+ 行)。无需手写 CRUD、序列化、SQL。
~95%
🔍
API 发现成本
update_name()、with_name_is()、execute_for_list() 命名一致，Agent 只需读一次 entity.rs，不再猜方法名。
~60%
🛡️
审计样板
.purpose("原因").comment("说明") 代替手写日志、监控、审计代码。
~80%
🗄️
Schema & 迁移
service_runtime_from_env() 自动处理建表 + 种子数据 + 连接池。零迁移脚本。
~90%
💬
上下文压缩
lossless-claw 将 10,000+ token 对话压缩为结构化摘要，保留决策和代码变更。
~70%
🐛
错误恢复
生成的 entity.rs 防止幻觉方法调用。平均修复迭代：1 次（修了个变量名）。
~50%

整体估算：本次构建相比手写 Rust 业务逻辑服务，总计节省约 75–85% 的 Token 消耗。

TeaQL 让 Agent 把 Token 花在「业务意图」上，而不是「样板代码」上。

🔍透明追踪 & 审计

TeaQL 的审计系统核心原则：每一次数据库操作都应该可解释。从业务意图到 SQL 执行，整条链路透明、可审查。

🗺️ 审计链路
意图 → 查询 → 持久化 → 追踪
.purpose("列出活跃学校") → .comment("按状态过滤") → .execute_for_list() → SQL Traces
操作人 → 操作 → 保存 → 审计记录
audit_as("管理员注册学校") → .save() → audit_data 表插入 → 变更历史保存
📂 本报告的证据链

每一层都保留了原始数据，支持完整的溯因分析：

🧠 提示
raw/agent-prompt.md — 完整任务指令
🤖 响应
raw/agent-response.md — Agent 决策过程
🔨 构建
raw/build-log.txt — cargo 完整输出
▶️ 运行
raw/runtime-log.txt — 10 步演示输出
📝 代码
raw/code-diff.patch — 生成 + 手写代码
🔎 可以回答的问题
  • Agent 有没有遵守 AGENTS.md 规则？→ 查 runtime 日志中的 purpose/comment
  • Agent 有没有猜方法名？→ 对比 entity.rs 与 main.rs 的调用
  • Agent 有没有幻觉 API？→ 查 build 日志中的编译错误
  • 审计链路是否完整？→ 查 .audit_as() 调用点

TeaQL 评估报告 #0072026.06.15
💡核心洞察
Agent 在 TeaQL 框架下可以非常有效

只要给出项目专属文档和明确的约束规则。AGENTS.md 规则链（刷新 CLI → 读 entity.rs → purpose/comment → audit_as → 不编辑生成文件）形成了一套 自我纠错护栏，能在 bug 进入生产之前拦截。

年迈硬件 + 现代框架 = 实用

在一台 2015 年的 MacBook Pro 上，从零构建冷编译仅需 1 分 17 秒，Agent 代码在编译通过后第一次运行即全部通过。验证了这套工作流在普通开发者设备上的实用性。

TeaQL 为 Agent 评估而生

设计特性 → 评估优势

  • .purpose() / .comment() — 查询语义可审查
  • 生成 API — 命名一致性消除幻觉
  • .audit_as() — 变更可追溯
  • SQL trace — 生成的 SQL 可取证
  • KSML + codegen — Agent 在边界内工作
