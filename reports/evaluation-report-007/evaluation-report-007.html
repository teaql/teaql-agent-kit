Token Savings: Where TeaQL Reduces Token Waste

One of TeaQL's core design principles is energy awareness — avoiding unnecessary consumption of computation, attention, and tokens. In AI agent evaluation, this means measurable token savings across the development lifecycle.

Boilerplate Elimination
24 Rust source files (3500+ lines) generated from a single 50-line KSML model. No hand-written CRUD code, no manual serialization, no hand-crafted SQL.
~95%
API Discovery
Consistent method names — update_name(), with_name_is(), execute_for_list(). Agent reads entity.rs once, never re-guesses.
~60%
Audit Boilerplate
Single .purpose("why").comment("what") chain replaces hand-written logging, monitoring, and audit instrumentation.
~80%
Schema & Migration
service_runtime_from_env() auto-handles schema creation, seed data, connection pooling — zero migration scripts.
~90%
Context Compression
Lossless-claw compaction reduces 10,000+ tokens of conversation to structured summaries preserving only decisions and code changes.
~70%
Error Recovery
Generated entity.rs prevents hallucinated method calls. Average error-to-fix: 1 iteration (a variable name).
~50%

Total estimated token savings across the full A-003 build: approximately 75–85% vs. writing equivalent Rust business service code by hand.

TeaQL makes the agent spend tokens on business intent instead of boilerplate mechanics.

Interpretation & Evidence Chain
Core Interpretation

Coding agents can be highly effective in TeaQL-based engineering work — given project-specific guidance, clear docs, and enforced rules. The AGENTS.md rule chain (refresh CLI → read entity.rs → purpose/comment → audit_as → never edit generated files) created guardrails that caught mistakes before production.

On a 10-year-old Intel MacBook Pro, the entire build completed in under 2 minutes from cold cache, and the agent's code passed all 10 runtime steps on first execution after compilation.

TeaQL Relevance
  • Explicit business intent — purpose() + comment() make query semantics inspectable.
  • Generated domain APIs — consistent method naming eliminates API hallucination vectors.
  • Audit trails — audit_as() ties every mutation to a description.
  • Runtime traceability — TEAQL_SQL env vars expose generated SQL.
  • Framework discipline — KSML + codegen ensures agent works within boundaries.
